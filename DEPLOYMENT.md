# Развёртывание моста как systemd-сервиса

Этот документ описывает, как мост запускается в постоянную работу под systemd с
автозапуском при загрузке и автоперезапуском при сбое.

## Почему режим `--listen` (критично)

Мост умеет три режима (см. `README.md`):

| Команда | Что делает |
|---|---|
| `bridge.py` (без флага) | **снапшот** → затем слушать поток |
| `bridge.py --snapshot` | только **снапшот** и выход |
| `bridge.py --listen` | **без снапшота**, сразу слушать поток AMQP |

**Снапшот чистит таблицы и делает purge очереди**, затем заново заливает состояние из
Redis. Но результаты исходов (`outcomes.status=4`, `result`) приходят **по AMQP после
завершения матча** — в Redis-снапшоте их нет. Поэтому снапшот при каждом рестарте
**затирал бы уже рассчитанные исходы**.

Из `bridge.py`:

```python
do_snapshot = not args.listen          # с --listen → False
...
if do_snapshot:
    Snapshotter(bridge).run()          # ← чистит таблицы + purge очереди + заливает из Redis
if do_listen:
    if not do_snapshot:
        bridge.warm_caches()           # только ЧИТАЕТ БД (прогрев ID-кэшей)
    Listener(bridge).run()             # только применяет апдейты из AMQP
```

`Snapshotter` (очистка) вызывается **исключительно** при `do_snapshot=True`. В сервисе
`ExecStart=… bridge.py --listen` → `args.listen=True` → `do_snapshot=False` → **очистка БД
не выполняется никогда** при автоперезапуске. Сервис при рестарте только прогревает кэши
(read-only) и продолжает слушать поток; накопленные результаты сохраняются.

Вывод: **сервис обязан работать в `--listen`.** Снапшот — отдельная одноразовая операция.

## Установка

1. Разместить мост в `/opt/feed-mysql-bridge`, создать `config.json` из
   `config.example.json` (доступы Redis/RabbitMQ выдаёт оператор фида), создать таблицы:
   ```
   mysql -u <user> -p <database> < schema.sql
   ```
2. **Первичная загрузка БД — один раз** (заливает состояние линии из Redis):
   ```
   python3 bridge.py --snapshot
   ```
3. Установить и запустить сервис:
   ```
   cp deploy/feed-bridge.service /etc/systemd/system/
   systemctl daemon-reload
   systemctl enable --now feed-bridge
   ```

С этого момента мост работает постоянно в режиме `--listen`, поднимается при загрузке
системы и автоматически перезапускается при сбое (`Restart=always`, `RestartSec=5`).

## Единственный консьюмер

На одну очередь RabbitMQ должен слушать **ровно один** экземпляр моста. Если запущены
несколько (например, забытые ручные `bridge.py` в `screen`), RabbitMQ раздаёт сообщения
между ними round-robin — состояние БД получается несогласованным. Перед запуском сервиса
убедитесь, что других процессов нет:

```
ps -eo pid,args | grep -F "bridge.py" | grep -v grep
screen -ls            # закрыть забытые сессии: screen -S <name> -X quit
```

## Эксплуатация

```
systemctl status feed-bridge          # состояние
journalctl -u feed-bridge -f          # живой лог (подключение, обработка, ошибки)
systemctl restart feed-bridge         # перезапуск БЕЗ очистки БД (режим --listen)
python3 db_info.py                    # размеры таблиц и разбивка по статусам
```

Проверка, что мост реально потребляет очередь (через RabbitMQ Management API): у очереди
должно быть `consumers = 1` и ненулевой `ack rate`.

> ⚠️ Единственное, что **очистит БД**, — ручной запуск `bridge.py --snapshot` или
> `bridge.py` без флага. Делать это только осознанно при первичной/повторной полной
> загрузке. Сервис (`--listen`) этого не делает.
